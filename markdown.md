# 项目核心总结

## 一、项目概述
本项目是**纯原生HTML/CSS/JavaScript实现的2D格斗游戏Demo**，核心目标是验证“用浏览器原生API手搓专业级打击感游戏”的可行性。通过零依赖开发，深入理解游戏引擎底层逻辑（如主循环、物理引擎、状态机），为后续进阶（Canvas重构、OOP架构）奠定基础。项目已部署至GitHub Pages（`https://<用户名>. github.io/ <仓库名>/`），实现全球可访问。


## 二、核心功能与技术实现
### 1. 游戏主循环与渲染系统
- **驱动机制**：基于`requestAnimationFrame`实现60fps主循环，分离**逻辑帧**（状态更新、碰撞检测）与**渲染帧**（DOM/CSS更新），避免卡顿。
- **坐标系统**：手搓绝对坐标计算，解决DOM相对定位与JS碰撞检测的“坐标割裂”问题（如CSS的`right`/`left`偏移同步）。

### 2. 物理引擎
- **基础物理**：实现重力（`velocityY -= gravity`）、摩擦力（`velocityX *= friction`）、击退，模拟真实角色运动。
- **击退系统**：命中时修改敌人速度向量，配合摩擦力实现“急停-滑行”效果，增强打击感。

### 3. 状态机与输入控制
- **状态管理**：用标志位（`isAttacking`、`enemyAlive`）实现**有限状态机（FSM）**，严格控制状态互斥（如攻击时无法移动，死亡后无法被攻击）。
- **输入处理**：使用监听器监听键盘输入，根据监听到输入的wasd、j、空格等按键变化，进行状态变化(例如`isright`),将控制角色的实际效果代码放到`gameloop()`函数里面,这样代码方便管理和维护

### 4. 打击感三件套
- **顿帧**：命中时冻结主逻辑60ms（`setTimeout(() => { /* 恢复逻辑 */ }, 60)`），制造“力量停滞”感。
- **屏幕震动**：用CSS `translate3d`强制GPU加速，配合JS触发`@keyframes`高频位移（如`document.body.style.transform = 'translateX(2px)'`），模拟冲击波。
- **视觉反馈**：命中时角色变色（`enemy.classList.add('...')`）、血条减少（`enemyHpFill.style.width = ...`），强化“击中”的即时反馈。


## 三、关键问题与解决方案（面试重点）
### 问题1：方向转换导致碰撞框错位
- **现象**：角色翻转（`scaleX: -1`）时，拳头碰撞框仍停留在原方向（如角色朝左时，拳头在右侧，无法命中敌人）。
- **根源**：CSS的`right: -50px`偏移是**相对父级**的，而JS碰撞检测需要**绝对坐标**，手动计算子元素坐标易出错。
- **解决方案：父子关系分离碰撞体**
  - **结构设计**：角色为父元素（`position: relative`），拳头为子元素（`position: absolute`），拳头通过`right: -50px`（或`left: 50px`）相对父级定位。
  - **自动同步**：父元素翻转时，子元素（拳头）通过CSS继承自动翻转（`transform: scaleX(-1)`），无需手动调整拳头坐标。
  - **精准检测**：碰撞检测直接获取拳头的`getBoundingClientRect()`，判断拳头的左右边与敌人边界是否重叠（如`fist.right > enemy.left && fist.left < enemy.right`），实现“视觉拳头”与“判定框”的1:1绑定。

### 问题2：“鞭尸”Bug（死亡后仍触发碰撞）
- **现象**：敌人血条归零、死亡动画播放中，仍能触发攻击判定（顿帧、震动）。
- **根源**：CSS动画（如`display: none`）是“视觉欺骗”，逻辑坐标（`enemyAlive`）未及时更新，导致碰撞检测仍在执行。
- **解决方案：数据与表现分离**
  - **逻辑先行**：在血量归零的第一时间，将`enemyAlive = false`（而非等待动画结束），切断所有后续碰撞检测（`if (!enemyAlive) return;`）。
  - **表现滞后**：死亡动画（`enemy.classList.add('...')`）在逻辑状态变更后执行，确保“视觉死亡”与“逻辑死亡”同步。


## 四、技术亮点与思考（面试加分项）
### 1. 底层原理理解
- **DOM/CSS与JS的协同**：深刻理解“CSS是表现层，JS是逻辑层”，避免“用CSS动画替代逻辑状态”（如死亡后仍能碰撞）。
- **父子关系的优势**：利用CSS的`position`继承特性，减少手动计算子元素坐标的复杂逻辑，提升代码可维护性。

### 2. 性能优化意识
- **AABB算法优化**：采用轴对齐包围盒（AABB）进行碰撞检测，减少计算量（仅判断矩形重叠，而非像素级检测）。
- **状态互斥**：用标志位锁死状态（如`isAttacking`），避免“同时执行多个状态”（如攻击时移动），降低逻辑复杂度。

### 3. 项目部署能力
- **GitHub Pages部署**：独立完成代码上传、分支配置（`main`分支根目录）、页面生成，展示全栈能力（前端开发+部署）。
- **静态资源管理**：确保所有静态文件（HTML/CSS/JS/图片）正确上传，避免“404”等部署问题。


## 五、项目价值与收获
- **证明原生能力**：通过纯原生开发，展示对浏览器底层API的掌握（如`requestAnimationFrame`、`getBoundingClientRect`），区别于“只会用框架”的候选人。
- **解决问题的思维**：面对“碰撞框错位”“鞭尸”等问题，能定位根源（视觉与逻辑割裂），并选择“父子关系分离碰撞体”“数据与表现分离”等有效方案，体现技术判断力。
- **进阶潜力**：项目为后续学习Canvas（性能优化）、OOP（架构升级）、指令缓存（连招系统）奠定基础，展示持续学习的能力。
